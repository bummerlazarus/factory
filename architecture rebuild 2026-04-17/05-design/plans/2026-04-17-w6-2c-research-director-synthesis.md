# W6.2c — Research-director synthesis Edge Function (plan)

**Date:** 2026-04-17
**Parent plan:** [2026-04-17-w6-2-research-director-split.md](2026-04-17-w6-2-research-director-split.md)
**Depends on:** W6.2a (theme scan) 🟢, W6.2b (connection scan) 🟢
**Downstream:** W6.2d (`/research` dashboard renders synthesized `body`)

---

## 1. Problem

W6.2a and W6.2b emit `reference_docs` rows of `kind='theme'` / `kind='connection'` with machine-generated titles and bodies. Example body:

```
Auto-generated by Research-director (W6.2a).

Window:   2026-04-16 22:30 UTC -> 2026-04-17 22:30 UTC
Baseline: 2026-04-15          -> 2026-04-16
Members: 3
Sources: csv, markdown
Namespaces: knowledge
Density: 2.00

## Members
- | : Opportunity | : "Parish minister training"...
- | Parish staff ministers (lay) | ...
```

`/research` (W6.2d) truncates the `body` to a 2-line preview. A raw dump like this isn't actionable — Edmund wants a sentence that says **"You've got three captures about parish-minister pricing — worth a one-page offering sheet?"**

This epic adds an Edge Function that reads unresolved rows, calls OpenAI, and rewrites `title` + `body` into a short human-readable proposal.

## 2. Input contract

```
POST /functions/v1/research-director-synthesis
Header: x-capture-secret: $CAPTURE_SECRET
Body:
  {
    "ids":   [uuid, uuid, ...]   // optional — target specific rows
    "window_days":  7             // optional — default 7 (how far back to scan)
    "limit":        20            // optional — hard cap on how many rows to synthesize
    "force":        false         // optional — re-synthesize even if synthesis_version set
    "model":        "gpt-4o-mini" // optional — default "gpt-4o-mini"
  }
```

Response:

```
200 {
  "processed": 5,
  "synthesized": 4,
  "skipped": 1,
  "failed": 0,
  "items": [
    { "id": uuid, "kind": "theme", "status": "synthesized",
      "new_title": "...", "tokens": {in: 412, out: 84} },
    ...
  ],
  "warnings": []
}
```

Errors: 401 on bad secret, 500 if `OPENAI_API_KEY` missing, 400 on bad body.

## 3. Auth

`x-capture-secret` header vs `CAPTURE_SECRET` env — matches every other Edge Function in this stack (capture, signals-ingest, youtube-ingest). `verify_jwt=false` at deploy so pg_cron / Shortcuts can hit it with just the shared secret.

## 4. Row selection

**Default scan** (no `ids` supplied):

```sql
SELECT id, slug, kind, title, body, metadata
FROM public.reference_docs
WHERE kind IN ('theme', 'connection')
  AND created_at >= now() - make_interval(days => $window_days)
  AND metadata->>'synthesis_version' IS NULL
  AND metadata->>'approved_at' IS NULL
  AND metadata->>'dismissed_at' IS NULL
ORDER BY created_at DESC
LIMIT $limit;
```

**Explicit `ids` supplied:** filter by id only (still respects `force` for the version check). Missing rows return as `status=not_found`.

**`force=true`:** drops the `synthesis_version IS NULL` filter so re-runs overwrite.

## 5. OpenAI call

- Model: **`gpt-4o-mini`** default (keeps cost at fractions of a cent per row). Overridable in body for A/B tests.
- Temperature: `0.4` (short, structured, low variance).
- `response_format: { type: "json_object" }` so we never parse markdown fences.
- Prompt per-row (see §6).
- Expected output:

```json
{
  "title": "Parish-minister pricing signals cluster",
  "body":  "Three captures in the last week circle around parish-minister coaching entry fees. Worth a one-page offering sheet?",
  "action": "Draft a short offering sheet"
}
```

- `body` must be 1-3 sentences. Prompt enforces this.
- Full OpenAI JSON stored in `metadata.synthesis_json` for debugging / future richer UIs.
- `body` column gets the plain text. `title` gets the new title (fallback to old on missing).
- Timeout: 20s per row. On failure: row skipped, `status=failed` in response, original `body` untouched.

## 6. Prompts

Two system prompts, one per kind — themes cluster many members; connections pair two items.

### Theme prompt

```
You are a research director synthesizing clusters found in a creator's corpus.

You'll be given a theme cluster: 2-10 items that our scanner grouped by
semantic similarity. Each item has a source (csv, markdown, capture,
youtube, conversation, etc) and a short snippet.

Your job: return ONE JSON object with:
- title:  a short noun-phrase title (under 70 chars). No quotes, no emoji.
- body:   1-3 sentences. State what threads these items share, AND what
          the creator might do about it (one concrete verb like "write",
          "record", "pitch", "reply to"). Total under 280 chars.
- action: optional 3-6 word imperative action phrase, or null.

Write like a thoughtful editor, not a marketer. Don't hype. If the items
look like near-duplicates of the same source text, say so plainly.
```

### Connection prompt

```
You are a research director synthesizing a cross-silo connection found
in a creator's corpus.

You'll be given two items from different sources (e.g., a captured note
and a podcast transcript) that our scanner matched above a similarity
threshold. Each item has a source, a namespace, and a short snippet.

Your job: return ONE JSON object with:
- title:  a short noun-phrase title (under 70 chars) naming the overlap.
- body:   1-3 sentences naming the shared idea AND proposing one concrete
          step (e.g., "turn the note into a podcast cold open"). Total
          under 280 chars.
- action: optional 3-6 word imperative action phrase, or null.

Don't hype. If the similarity looks coincidental (same boilerplate, same
author), say so plainly in the body.
```

### User content (per row)

Constructed from `metadata` + `body`:

- **Theme:** dump member snippets (`metadata.member_memory_ids` → look up short content from `memory.content[:300]` for each), member sources, member namespaces, cluster size.
- **Connection:** dump both items' snippets (via `metadata.a_memory_id` / `metadata.b_memory_id`), both sources, both namespaces, similarity score.

Snippet size capped to 300 chars/member so we stay well under input token budgets.

## 7. Metadata write

On success:

```sql
UPDATE public.reference_docs
   SET title = $new_title,
       body  = $new_body,
       metadata = metadata || jsonb_build_object(
         'synthesis_version',  1,
         'synthesized_at',     now()::text,
         'synthesis_model',    'gpt-4o-mini',
         'synthesis_json',     <full OpenAI JSON>,
         'original_title',     <previous title>,   -- preserve for debug
         'original_body',      <previous body>
       ),
       version = version + 1
 WHERE id = $row_id;
```

- `synthesis_version = 1` signals "synthesized at least once". Re-run with `force:true` bumps to 2, 3, …
- `synthesized_at` = ISO timestamp. /research can show "Synthesized 2h ago" badge if useful later.
- `original_title` / `original_body` preserved on first synthesis only (don't overwrite on re-runs — keeps the raw SQL-generated version available for reference).

## 8. How /research renders this

`app/research/page.tsx` reads:
- `row.title` — rendered as the card heading
- `row.body` — `line-clamp-2` truncated preview

So after synthesis, `body` is a 1-3 sentence readable proposal. `metadata.synthesis_json.action` is available for future "quick actions" polish but the v1 UI doesn't consume it. No W6.2d code changes needed.

## 9. Cost

- ~500 input tokens + ~100 output tokens per row on `gpt-4o-mini`.
- $0.15/1M input + $0.60/1M output → $0.000075 + $0.00006 = **~$0.000135/row**.
- 20 rows/week → ~$0.003/week. Annual: ~$0.15.
- `limit` cap + `synthesis_version` idempotency keeps this bounded even on a wild day.

Well under the $2 epic ceiling; trivial compared to the $10/run project ceiling.

## 10. Scheduling

- Monday 07:00 UTC: `research-director-weekly` cron runs `research_director_themes()` + `research_director_connections()`.
- Monday 07:05 UTC: synthesis should run.

Options:

### A. `net.http_post` from pg_cron (preferred)

Enable `pg_net` extension (already available, not installed). Add a second cron job:

```sql
SELECT cron.schedule(
  'research-director-synthesis-weekly',
  '5 7 * * 1',
  $$
    SELECT net.http_post(
      url := 'https://obizmgugsqirmnjpirnh.functions.supabase.co/research-director-synthesis',
      headers := jsonb_build_object(
        'Content-Type',      'application/json',
        'x-capture-secret',  current_setting('app.capture_secret', true)
      ),
      body := jsonb_build_object('window_days', 7, 'limit', 50)
    );
  $$
);
```

Complication: `CAPTURE_SECRET` isn't in pg settings. Options: (a) set `app.capture_secret` via Vault, (b) store secret in a private `function_secrets` table and SELECT it at cron time, (c) hardcode the secret in the cron command (simplest, acceptable because `cron.job` is in the `cron` schema which isn't exposed to anon).

Pick **(c)** hardcode — matches how other projects bootstrap pg_net jobs. Document clearly. Can refactor to Vault later.

### B. Second cron command chained in existing job

pg_cron can run multi-statement commands. Append a second SELECT that calls `net.http_post` — same one-job approach as W6.2b chained onto W6.2a.

### Decision

**B (chain onto existing `research-director-weekly`)**. Keeps all weekly synthesis in one cron row. Synthesis runs immediately after connections scan completes, no 5-minute gap needed (the scans are fast). If that gets unwieldy later, split out.

Requires: `CREATE EXTENSION IF NOT EXISTS pg_net;` as part of this epic.

## 11. Verification plan

1. **Deploy** function; confirm listed via `list_edge_functions`.
2. **Seed** one `theme` row and one `connection` row via direct INSERT — set bodies to the exact machine-generated format so the synthesis prompt has realistic input.
3. **Call** function with `{ids: [theme_id, connection_id]}` + secret.
4. **Assert:**
   - HTTP 200, `synthesized=2`
   - `reference_docs.title` for each row changed to readable title
   - `body` is 1-3 sentences, <280 chars
   - `metadata.synthesis_version = 1`
   - `metadata.synthesized_at` is set (ISO timestamp)
   - `metadata.original_body` preserved
5. **Idempotency:** call again without `force`. Assert `skipped=2`, no version bump.
6. **Real-data pass:** call with no `ids`. Expected: 0 rows (corpus just got cleaned up). Log it.
7. **pg_net + cron:** `CREATE EXTENSION pg_net`, update cron command, `SELECT cron.schedule()` replaces existing job, verify new command contains synthesis call.
8. **Cleanup:** delete seeded rows.
