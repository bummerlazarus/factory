## Goal

Daily job that clusters recent `observations` by embedding cosine similarity and writes one cluster summary per connected component to `public.reference_docs` (kind = `cluster`). Inputs for:
- **W6.2** Research-director Skill (weekly cross-corpus scan тАФ will query these clusters)
- **W6.3** IP map doc (living doc тАФ will be built from these clusters plus `reference_docs`/`skill_versions`)

## Key design choices

### Option chosen: **A тАФ pg_cron + SQL function**

`pg_cron` 1.6.4 was available on the project (not yet installed). Installed it in the same migration. Function runs entirely in Postgres тАФ no Edge Function, no external scheduler, no egress.

### How observations reach an embedding

`observations` has no `embedding` column and no FK to `memory`. The join path is the existing pairing convention:

```sql
JOIN memory m ON m.source = 'observation' AND m.source_id = observations.id::text
```

This is how W1.1 paired captured-data rows to their embeddings across other source types (see `memory.source, memory.source_id` with the unique `(source, source_id)` index). The Librarian re-uses it. No re-embedding happens тАФ the clustering only considers observations that already have a `memory` row.

**Implication for upstream:** whatever pipeline writes `observations` needs to also write a matching `memory` row with `source='observation', source_id=<obs.id::text>`. Today nothing does, so the window is empty. The function returns cleanly in that case.

### Clustering algorithm (deliberately dumb)

1. Pull all `observations` in window that have a matching `memory` embedding. Cap at 500 most recent.
2. Pairwise distances via `pgvector`'s `<=>` (cosine distance). Edge exists when `distance < 1 - 0.85 = 0.15`, i.e. cosine similarity > 0.85.
3. Undirected adjacency, recursive CTE produces connected components (keyed by `min(node)`).
4. Drop singletons (`count >= 2`).
5. For each surviving component, pick exemplar = highest edge degree, tiebreak newest.
6. Render title + bulleted body, INSERT to `reference_docs`.

No k-means, no HDBSCAN, no cluster-centroid math. If a cluster has more than a handful of members this will still work тАФ the connected-components semantics means even a "chain" of loosely similar nodes gets grouped, which is the intended behavior for "topic drift within a week."

### `reference_docs` fit

W5.2 established `slug`/`kind`/`body`/`metadata`. Added `'cluster'` to the `kind` CHECK constraint. Nothing else was added.

- `slug` = `cluster-<YYYY-MM-DD>-<exemplar_id_prefix8>` тАФ deterministic, so a same-day re-run upserts rather than duplicates.
- `title` = `Cluster <date> тАФ <first 80 chars of exemplar body>`.
- `body` = a short header + bulleted list of the 200-char snippets of member observations.
- `metadata` = `{window_start, window_end, cluster_size, member_observation_ids[], exemplar_observation_id, similarity_threshold, generator, generated_at}`.

### Idempotency

`INSERT ... ON CONFLICT (slug) DO UPDATE SET title, body, metadata, version = version+1`. Safe to re-run at any time.

### Schedule

`SELECT cron.schedule('librarian-daily', '0 6 * * *', $$ SELECT public.librarian_cluster_observations(7); $$)`. 06:00 UTC daily, 7-day window.

## Zero-observation handling

First statement in the function after the JOIN: `IF v_processed = 0 THEN RETURN jsonb_build_object(...) END IF`. No empty tables, no NULL propagation, no failed aggregates. Verified by calling the function twice today with the live (empty) data тАФ both returns looked like:

```json
{"processed": 0, "clusters_written": 0, "note": "no observations with embeddings in window", ...}
```

## Smoke test (positive path)

- Inserted 3 synthetic observations (kind='pattern', the allowed enum value).
- Inserted 3 paired `memory` rows. Two got the same template embedding; one got an unrelated embedding from a different corpus.
- Ran `librarian_cluster_observations(1)`. Got `{processed:3, clusters_written:1}`.
- Inspected `reference_docs WHERE kind='cluster'` тАФ one row, `cluster_size=2`, `members_n=2`, the orphan was correctly dropped.
- Re-ran the function тЖТ same row, `version` bumped to 2, count still 1. Upsert works.
- Deleted the test observations, paired memory rows, and the cluster row.

## Open follow-ups (not in this ship)

1. **Upstream:** something needs to start writing `observations` rows and paired `memory` rows with `source='observation'`. Today the table is empty (total 0). Likely candidates: the `capture()` Edge Function could emit observations when it detects a "pattern" (todo-like, risk-like, etc.), or a Skill could summarize sessions into observations and embed them.
2. **Exemplar title is crude.** For cluster sizes that grow, "first 80 chars of exemplar body" will read weirdly. W6.2 / W6.3 can replace the title with an LLM synthesis when they're ready тАФ just UPSERT on the same slug and the librarian's version will bump.
3. **Cluster decay.** Old `cluster` reference_docs rows are not expired. At 1/day with small input this is fine for months. If it becomes noisy, add a maintenance task that archives clusters whose `metadata.window_end < now() - interval '90 days'`.
4. **Threshold 0.85 is an untested guess.** Revisit once there's real observation volume.
