## Finding

**File browser is NOT migrated.** `lib/files.ts` is still filesystem-based reading from `COWORK_PATH` (iCloud). This blocks Vercel production deploy тАФ serverless functions can't reach an iCloud path.

## Concrete inventory

- **`dashboard/lib/files.ts`** (156 lines) тАФ `import fs from "fs/promises"`, `process.env.COWORK_PATH`, `process.cwd() + "/docs"`. Functions: `getFileTree`, `readFile`, `writeFile`, `searchFiles`, `getDocsTree`, `readDocsFile`, `writeDocsFile`.
- **`dashboard/app/api/files/route.ts`** тАФ imports from `@/lib/files`. The `/api/files` JSON endpoint.
- **`dashboard/app/api/files/raw/route.ts`** тАФ separate raw-file endpoint, same shape.
- **`dashboard/lib/tools.ts`** тАФ also references `COWORK_PATH` (agent tool bindings).

So migrating /files is not a single file change тАФ it's the file-reading surface that the agent tool layer also uses.

## Not blocking anything downstream except

Vercel production deploy (W3.9). Dashboard runs fine locally.

## Three options for W3.7 execution

### (a) Mirror iCloud тЖТ Supabase Storage with an ongoing sync
- Pros: Works on Vercel. Preserves the filesystem view Edmund is used to.
- Cons: Sync loop (watch iCloud, upload changes) тАФ custom infra. Storage cost scales with COWORK size.
- Shape: Local daemon or CI job that diffs COWORK_PATH and pushes to a Storage bucket. `/files` reads from bucket.

### (b) Disable /files in production (dev-only feature)
- Pros: Zero migration. Vercel ships the dashboard without the browser.
- Cons: Edmund loses file browsing in production. Contradicts pillar 1 (rich data capture + single surface).
- Shape: Guard `/files` + `/api/files*` behind `process.env.COWORK_PATH` presence; 404 in production.

### (c) Hybrid тАФ local when COWORK_PATH set, Supabase Storage otherwise
- Pros: Best of both. Dev still reads local iCloud. Prod falls back to Storage.
- Cons: Two code paths. Data needs to live in Storage too (so someone/something uploads).
- Shape: Runtime branch in `lib/files.ts` on `COWORK_PATH`. Upload sync can be manual or scheduled.

## Recommendation

Option **(c) Hybrid** with one caveat: in v1, the Storage side can be **empty** тАФ `/files` just shows an empty tree in prod. That ships Vercel deploy without a content migration. Edmund adds files to Storage over time as the agent system needs them; eventually the iCloud COWORK path becomes redundant and we drop it.

Smaller surface than (a), doesn't lose the feature like (b).
