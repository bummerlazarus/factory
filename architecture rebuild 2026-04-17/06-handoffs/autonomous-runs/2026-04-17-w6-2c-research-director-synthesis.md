## 6. Design notes

- **`response_format: json_object`** — skipped markdown-fence wrangling (signals-ingest has a `stripFences()` helper; not needed here).
- **Secret hardcoded in cron command** — per plan §10. The `cron.job` table lives in the `cron` schema which is not exposed to `anon`/`authenticated` roles, so this is safe. Refactor to Vault-backed `current_setting()` later if multi-environment.
- **Missing member memory fallback** — if `member_memory_ids` don't resolve (e.g., memory rows deleted since scan), synthesis falls back to the machine-generated body as the user message. Prevents hard failure on delete-after-scan.
- **Skip approved/dismissed rows** — besides `synthesis_version`, also short-circuits on `metadata.approved_at` or `metadata.dismissed_at`. No point rewriting triaged rows.
- **W6.2d untouched** — `/research` already renders `body`. No UI change needed; the synthesized text slots into the existing 2-line preview.

## 8. Follow-ups

1. **Vault-backed secret for cron.** Hardcoding the `CAPTURE_SECRET` in `cron.job.command` works but isn't rotatable. When Edmund rotates the secret, the cron needs `SELECT cron.schedule()` re-applied. Candidate: `supabase secrets` + `current_setting('app.capture_secret')` via a startup GUC, or `vault.secrets`. Low priority — single-env today.
2. **W6.2 parent epic.** With W6.2a/b/c/d all shipped, flip W6.2 itself 🔵 → 🟢 in the backlog (out of scope for this run — updating only W6.2c per task spec).
3. **Consider `action` pills in `/research` UI.** `metadata.synthesis_json.action` is stored but the v1 UI doesn't render it. Cheap W6.2d polish: one quick-action button per card.
4. **Observability.** `net._http_response` only keeps recent rows. For Monday's real cron run, checking whether synthesis fired correctly will mean either scanning `reference_docs.metadata->>'synthesized_at'` or dropping a line into `work_log`. Consider a tiny "cron heartbeat" row for visibility.
5. **Token budget for very large themes.** Current snippet cap is 300 chars × 10 members = 3000 chars max user prompt. On a 50-member theme (unlikely today), the prompt would be clipped. Fine for current corpus shape; revisit if we ever see 20+ member clusters.
