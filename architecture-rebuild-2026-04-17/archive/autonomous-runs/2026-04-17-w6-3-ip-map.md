### Migration 2 тАФ inline edit note

The version stored in `dashboard/supabase/migrations/20260417151000_ip_map_function.sql` uses `LATERAL regexp_matches(...) AS m(term_arr), LATERAL unnest(term_arr) AS term`. When I applied the migration via MCP I used an equivalent cleaner form that relies on the fact that `regexp_matches(..., 'g')` already returns one row per match with each row being `text[]` of length 1 тАФ so I access the match as `m[1]` and compute `terms` via a correlated subquery. Both forms are equivalent for our single-capture-group regex. The file mirror is the canonical text; the live function is the idiomatic PL/pgSQL form.

## 4. Gap-detection heuristic (picked & documented)

**A gap is EITHER:**
- **(a) Thin framework** тАФ `kind='framework'` AND `length(body) < 400` AND metadata lacks both `source_path` and `imported_at`. Catches speculative framework slugs never actually written up.
- **(b) Recurring theme head** тАФ among themes titled `'Theme YYYY-MM-DD тАФ <text>'` created in the last 30 days, the first word of the text-portion (lowercased) appears across `>= 2 distinct weeks`. Catches topics that keep surfacing without anyone authoring a framework.

**Why this heuristic?** We have no FK from `framework` тЖТ `workspace_items` yet (workspace_items is 0 rows today). So "no concrete artifact" is measured through body length + metadata hints тАФ an imperfect but easy-to-revise signal. Theme-head-word recurrence directly captures the "hinted at long enough to deserve authorship" pattern without needing semantic re-clustering.
