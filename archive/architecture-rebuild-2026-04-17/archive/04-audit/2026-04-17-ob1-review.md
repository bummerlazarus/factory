# OB-1 Architecture Review — For Edmund's Rebuild

**Date:** 2026-04-17
**Source:** `/Users/edmundmitchell/factory/research/reference-repos/OB1-main/`

## 1. What is OB-1?

Open Brain is a persistent AI memory infrastructure — a single Supabase database with pgvector, one OpenRouter embedding gateway, and an MCP protocol that any AI client (Claude, ChatGPT, Cursor) can plug into. Design philosophy: **"one database, one protocol, many clients"** — no middleware chains, no SaaS multipliers. Users build on top via curated **extensions** (household knowledge, meal planning, professional CRM, job pipelines) that compound as they connect.

### Key abstractions

- **`/schemas/`** — DB table *extensions* (new columns/tables alongside core `thoughts`). Example: `workflow-status` adds status tracking. No MCP server; just SQL.
- **`/primitives/`** — Reusable conceptual patterns or deployment templates. Examples: RLS, remote-mcp, shared-mcp, deploy-edge-function. Curated; must be used by 2+ extensions to justify extraction.
- **`/extensions/`** — Ordered learning path (6 builds). Each adds domain-specific tables + MCP tools + RLS rules. Extensions 1–3 use core `thoughts` table; 4–6 add domain tables.
- **`/recipes/`** — Standalone capabilities (importers, workflows, skill packs). No prescribed order.
- **`/integrations/`** — External connections, alternative deployments (Slack, Discord, K8s).
- **`/skills/`** — Reusable agent prompts. Install once, use anywhere.
- **`/dashboards/`** — Frontend templates (SvelteKit, Next.js).

## 3. Patterns worth stealing

### 3.1 Remote MCP via Supabase Edge Functions
**File:** `/primitives/remote-mcp/` + `/server/index.ts`

Deploy MCP servers as HTTP endpoints in Supabase Edge Functions. Connect from any AI client via URL + key. No local Node.js, no config files, no bridge processes.

**For Edmund:** Each venture (ZPM, Real+True, Faith&AI) can have its own Edge Function serving MCP tools, all behind the same Supabase project with different keys. Scales to many clients; no custom runtime. **Confidence: HIGH.**

### 3.2 Extensions as domain modules
**File:** `/extensions/` structure + `metadata.json`

Each extension is a self-contained folder: `README.md` + `schema.sql` + `index.ts` (MCP server) + `metadata.json`. No shared state between extensions except the core `thoughts` table and auth context. Metadata declares dependencies.

**For Edmund:** Ventures are orthogonal, not a learning path, but the **modularity pattern transfers directly**. Concrete layout:

```
/extensions/
  /zpm/            README.md + schema.sql + index.ts + metadata.json
  /real-true/
  /faith-ai/
  /client-work/
```

Makes setup reproducible and dependencies explicit. **Confidence: HIGH.**

### 3.3 RLS as the security boundary
**File:** `/primitives/rls/` + `/extensions/professional-crm/schema.sql`

RLS enforces multi-user isolation at the DB layer, not in app code. User-scoped (`auth.uid() = user_id`); household-scoped via junction tables; service-role bypass for Edge Functions.

**For Edmund:** Use RLS for any multi-user surface (client teams on Real+True, spouse access). Shared access doesn't bypass the DB — it gets a separate Edge Function + separate API key + scoped role. **Confidence: HIGH.**

### 3.4 Metadata.json for machine-readable dependencies
Every extension and recipe declares: name, category, author, version, required primitives, required services, learning order, estimated time, tags, difficulty.

**For Edmund:** Apply to ventures. `/extensions/zpm/metadata.json` declares `requires: ['supabase', 'pinecone']`, `requires_primitives: ['rls', 'shared-mcp']`, `requires_secrets: ['OPENROUTER_KEY']`. Makes setup auditable and CI-checkable. **Confidence: MED.**

### 3.5 Skills in the repo (not hidden in Claude)
Plain markdown workflows, versioned in the repo, installable into any AI client.

**For Edmund:** Store venture-specific skills in `/skills/<venture>/`. Skills declare which MCP tools and tables they depend on. **Confidence: MED-HIGH.**

## 4. Patterns to NOT adopt

### 4.1 Local compiled MCP servers
`/primitives/shared-mcp/index.ts` shows shipping a compiled Node.js MCP server to a spouse/team member's machine. Requires Node, `.env` files, config edits, upkeep. **This is outbuilding.** If sharing is needed, deploy as an Edge Function (remote), not on the user's machine.

### 4.2 Heavy custom middleware
K8s deployment (`/integrations/kubernetes-deployment/`), life-engine orchestration recipes, custom dashboard frameworks — valuable for OB-1's community but operational debt for Edmund. Stick to Supabase Edge Functions + Vercel.

### 4.3 Household-style multi-user models
Junction tables for `household_members` with complex RLS are overkill for solo/small-team ventures. Start with user-scoped RLS; add shared-MCP only when a specific venture needs team/client access.

## 5. Extensions-as-ventures: is it the right model?

**Yes — with modifications.**

What transfers:
- Folder-per-venture with `README.md` / `schema.sql` / `index.ts` / `metadata.json`
- Explicit dependencies via metadata
- Each venture owns its domain schema; cross-venture queries hit a shared table
- Each venture's Edge Function exposes tools scoped to its tables

What doesn't transfer:
- Learning order (ventures are parallel, not sequential)
- Household/family sharing patterns (most ventures are solo or small-team)
- OB-1's compounding narrative

## 7. Supabase usage patterns in OB-1

1. **pgvector + HNSW** — `thoughts.embedding` (1536-dim from OpenRouter). `match_thoughts()` RPC for semantic search with metadata filters.
2. **RLS enforcement** — `auth.uid()` scopes queries at DB layer. Service role bypasses for Edge Functions; scoped roles enforce for shared access.
3. **Edge Functions as runtime** — MCP servers as HTTP endpoints. Deno runtime, auto-scaling.
4. **JSONB metadata** — `thoughts.metadata` holds `{topics, type, people, action_items}`. GIN index for fast filter queries. LLM-populated at capture.
5. **`supabase secrets set`** — per-function API keys (different secrets per extension / shared server).
6. **Idempotent migrations** — `IF NOT EXISTS` prevents re-run errors.

Edmund should adopt all of these.

## 8. MCP patterns: remote vs. shared

| Pattern | When to use | Security model |
|---|---|---|
| **Remote MCP** (`/primitives/remote-mcp/`) | One user, multiple clients (Claude + ChatGPT + Cursor) | URL + API key; simple key-check |
| **Shared MCP** (`/primitives/shared-mcp/`) | Part of your brain shared with someone else (spouse, teammate, client) | Separate Edge Function + scoped DB role + RLS |

**For Edmund:** Every venture gets one remote MCP Edge Function (primary). If a venture involves team/clients, add a shared MCP with a scoped role. Store access keys as separate Supabase secrets (`MCP_ZPM_KEY`, `MCP_ZPM_SHARED_KEY`).

## Summary

OB-1 is a well-architected template. Steal these five things (extensions-as-modules, remote MCP via Edge Functions, RLS, skills-in-repo, metadata declarations). Skip these three (local compiled MCP servers, K8s/custom orchestration, household/family patterns).

The extensions + primitives + metadata model fits Edmund's target six-folder layout if organized **by venture instead of by learning order**. OB-1's remote-mcp + shared-mcp patterns are the cleanest way to wire multi-client, multi-venture access to a single Supabase + Pinecone backbone.
