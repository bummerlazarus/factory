## Phase 2 — Curator nudges

**Decision: pg_cron + pg_net, not local Claude Code scheduled task.**
The roadmap brief said "Claude Code Cron." But pg_cron + pg_net are already enabled, and a Postgres cron calling an Edge Function runs even when Edmund's laptop is off. Local Claude Code scheduled tasks die when the harness isn't open. Switching to pg_cron is more durable and matches the existing 024_table_registry_drift_cron.sql pattern.

**Decision: write to `skill_versions` directly, not to a new `inbox_items.kind`.**
Discovery during this build: `dashboard/app/api/promotions/route.ts` already approves/rejects rows in `skill_versions WHERE status='proposed'`. UI at `dashboard/app/inbox/promotions/page.tsx` already renders them. The approval loop exists end-to-end; it's just been waiting for proposers. Corva writes there.

**Decision: Anthropic API, not OpenRouter / multi-provider.**
Curator runs once a day. Cost is negligible. Don't add provider complexity for v1.

**Files:**
- Create: `supabase/migrations/028_curator_pass.sql` — pg_cron job that calls curator_pass via pg_net, plus a `curator_runs` log table.
- Create: `supabase/functions/curator_pass/index.ts` — Edge Function that fetches last 24h, calls Anthropic, writes proposed skill_versions rows.
- Modify: `ops/changelog.md`.

**Success criteria:**
- Manually invoking the function writes ≥1 skill_versions proposal that shows up at `/inbox/promotions`.
- Cron fires nightly at 3am local (8am UTC since Edmund is in Texas).
- `curator_runs` row written each pass with status + counts.

## Phase 3 — Skills self-improvement loop

**Discovery during this build: the loop is 80% built already.**
- `skill_versions` table: ✅
- Approve/reject UI + API: ✅ (`dashboard/app/api/promotions/route.ts`, `dashboard/app/inbox/promotions/page.tsx`)
- Phase 2 above writes the proposed rows: ✅
- **Missing**: when Edmund clicks Approve, the disk SKILL.md isn't updated. Notion isn't updated. The DB row flips to `status='approved'` but the actual SOP file on disk stays old.

**Decision: shell out from the Next.js approve handler to a node script that does disk + git.**
The dashboard runs locally on Edmund's laptop (launchd Phase 0). It has filesystem access. The cleanest way: API route writes a job marker, a separate `ops/bin/promote-skill.mjs` watches/runs. But simpler: `child_process.exec` the script directly from the API route. Since this is local-only (Vercel deploy doesn't matter for skills), that's fine.

**Decision: skip the Notion-write step for v1.**
Notion writes from a Next.js API route would require the Notion MCP, which isn't reachable from API code. Edmund manually mirrors high-value skills to Notion today; the loose-end ticket can promote that later. Disk + git is the meaningful win.

**Decision: skill_usage tracking is a NICE-to-have, defer it.**
The original Phase 3 plan added a `skill_usage` table to feed the curator. But the curator's first pass works fine on broader signals (work_log, ingest_runs, sessions). Add per-skill usage logging only after Edmund sees value in the curator's proposals.

**Files:**
- Create: `ops/bin/promote-skill.mjs` — reads a skill_versions row by id, writes `~/.claude/skills/<name>/SKILL.md` (and `~/Library/Mobile Documents/com~apple~CloudDocs/CEO Cowork/skills/` if it exists), commits.
- Modify: `dashboard/app/api/promotions/route.ts` — on approve, spawn `node ops/bin/promote-skill.mjs <id>` and capture output into the work_log artifact.
- Modify: `ops/changelog.md`.

**Success criteria:**
- Approve a proposed skill_version in the dashboard → disk file updated, git commit created in factory.
- Output of the script visible in dashboard logs (or surfaced in the work_log artifact).
- Reject does nothing on disk (only DB).

## Phase 4 — Multi-platform messaging gateway

**Decision: build the full 4-platform shape (Telegram, Slack DM, iPhone Shortcut, Email-webhook), but most platforms need Edmund's hand to provision.**

Edmund-required setup (NOT autonomous):
- **Telegram**: Edmund creates a bot via `@BotFather`, gets the token, sets the webhook URL. Token goes in `dashboard/.env.local` as `TELEGRAM_BOT_TOKEN`.
- **Slack DM**: requires creating a Slack app + adding bot scopes + installing to his workspace. Punt to a separate session unless he actively uses it.
- **iPhone Shortcut**: Edmund creates a Shortcut that POSTs to the gateway URL. Doc + curl examples make this 30 seconds for him.
- **Email**: needs an inbound-email service (SendGrid Inbound Parse, Mailgun routes, etc). Defer entirely — out of scope tonight.

**What I CAN build autonomously:**
- The unified `app/api/gateway/[platform]/route.ts` handler scaffolding
- A `gateway_threads` table that maps `(platform, platform_thread_id) → internal thread_id`
- Extension to `capture()` accepting `thread_id` + `platform` fields
- Setup doc with copy-paste BotFather instructions and webhook URL

**Decision: Telegram first (Edmund's most-mentioned channel), defer Slack/Email.**
Edmund's prior Telegram-on-Railway attempt failed; the rebuild is on Vercel (dashboard preview URL is publicly reachable, qualifies as a webhook target). Single platform first → prove the loop → add others.

**Files:**
- Create: `supabase/migrations/029_gateway_threads.sql` — `gateway_threads` mapping table.
- Modify: `supabase/functions/capture/index.ts` — accept optional `thread_id` and `platform` fields; persist on inbox_items.
- Create: `dashboard/app/api/gateway/telegram/route.ts` — webhook handler.
- Create: `dashboard/lib/gateway-normalize.ts` — Telegram payload → unified shape.
- Create: `ops/docs/gateway-telegram-setup.md` — Edmund's 5-minute setup walkthrough.
- Modify: `ops/changelog.md`.

**Success criteria (what I can verify autonomously):**
